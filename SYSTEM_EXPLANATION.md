# SurfSense — System Explanation

This document explains the evaluation design of the SurfSense thesis project in three parts:
1. What the scenarios are and how they were constructed
2. What the dataset contains, how it was split, and how missing values were handled
3. How the system was tested end-to-end — inputs, outputs, and evaluation metrics

---

## Part 1 — Scenarios

### What a scenario is

A **scenario** is not just a prompt. It is a **test case** that bundles three things together:

1. **A frozen forecast snapshot** — a JSON file in `scenarios/snapshots/` containing pre-fetched hourly weather and wave data for a specific spot and time window. Snapshots were created by calling the live Open-Meteo API once and saving the result. This means every evaluation run replays the same numbers — the scenario is deterministic even though both systems being compared use LLMs.

2. **A skill level** — `beginner`, `intermediate`, or `advanced`. This controls the safety thresholds applied to the forecast and the framing of the natural-language query.

3. **An automatically-derived user query** — a short natural-language request formatted from the spot name, date, skill level, and time window. For example: `"I'm a beginner surfer. Please check the surf conditions at Praia do Guincho on 2026-05-08. Rate each hour as ideal, suitable, challenging, or unsafe, flag any unsafe hours, and identify the best surf windows. No need to ask for further details."`

### How they were created

The scenario catalogue lives in `scenarios/scenarios.json` and is generated by `scenarios/generate_scenarios.py`.

The generation script defines a **coverage matrix**: every combination of snapshot × skill level is a scenario. The matrix is:

| Snapshot | Location | Window |
|----------|----------|--------|
| `guincho_24h` | Praia do Guincho, Portugal | 24 hours (spring) |
| `guincho_winter_24h` | Praia do Guincho, Portugal | 24 hours (winter) |
| `ericeira_5d` | Ericeira, Portugal | 5 days |
| `peniche_5d` | Peniche / Supertubos, Portugal | 5 days |
| `sagres_5d` | Sagres / Tonel, Portugal | 5 days |
| `hossegor_5d` | Hossegor, France | 5 days |
| `jeffreys_bay_5d` | Jeffreys Bay, South Africa | 5 days |

Each of the 7 snapshots is crossed with all 3 skill levels, giving **21 scenarios total**.

The Guincho spot appears twice (spring and winter) specifically to test whether the system behaves differently under genuinely different forecast conditions — the winter snapshot contains dangerous wave heights and very strong winds, while the spring snapshot starts calm and deteriorates during the day.

### What a snapshot contains

A snapshot is a structured JSON file. Each hourly entry contains:
- `timestamp` (UTC ISO string)
- `waves.avg_m`, `waves.min_m`, `waves.max_m` — wave height in metres
- `swell.height_m`, `swell.period_s`, `swell.direction`, `swell.direction_deg`
- `wind.speed_kph`, `wind.direction`, `wind.direction_deg`, `wind.is_offshore`, `wind.is_light`
- `summary` — a one-line text description for that hour

### What is being tested

The three "named" scenario scripts (`01_single_spot_guincho.py`, `02_multi_spot_trip.py`, `03_guincho_ml.py`) are **demonstration scenarios** used as thesis walkthroughs in Section 3.4. They are each driven by a single natural-language user message and exercise different parts of the pipeline:

- **Scenario 1** (beginner, Guincho, 24 h, rule-based scoring): tests the single-spot assessment path — `research_spot → fetch_forecast → assess_conditions → find_surf_windows`.
- **Scenario 2** (intermediate, Ericeira + Peniche + Sagres, 5 days, rule-based): tests the multi-spot itinerary path — the same chain runs three times in parallel and then a trip planner aggregates the results.
- **Scenario 3** (beginner, Guincho, 24 h, ML scoring): identical query to Scenario 1 but with `SCORING_MODE=ml`. The orchestrator is instructed to name the top SHAP contributors in its response.

The full 21-scenario catalogue is used by the **LLM baseline evaluation** (Section 3.5.2). There, the purpose is to compare SurfSense against GPT-4o-mini on the same frozen inputs and score both systems on five automated dimensions (see Part 3).

## Part 2 — The Dataset

### What it contains

The training dataset (`ml/data/processed/historical.parquet`) contains **87,720 hourly rows** across five geographically diverse surf spots, covering the period **2024-04-25 → 2026-04-25** (approximately two years). Each row is one hour at one spot.

The five spots are:

| Spot | Location | Coordinates |
|------|----------|-------------|
| Pipeline | Oahu, Hawaii | 21.67°N, 158.05°W |
| Hossegor | Aquitaine, France | 43.66°N, 1.42°W |
| Ericeira | Portugal | 38.96°N, 9.42°W |
| Jeffreys Bay | Eastern Cape, South Africa | 34.05°S, 24.92°E |
| Gold Coast | Queensland, Australia | 28.00°S, 153.43°E |

Each spot contributes exactly **17,544 rows** (24 hours × 365 days × ~2 years).

### Columns

Every row has these columns after collection and renaming:

**From the Open-Meteo Marine API (wave data):**
- `wave_height_m` — combined wave height
- `wave_period_s`, `wave_direction_deg`
- `swell_height_m`, `swell_period_s`, `swell_direction_deg`
- `wind_wave_height_m`, `wind_wave_period_s`
- `water_temp_c`

**From the Open-Meteo Archive API (atmospheric data):**
- `wind_speed_kph`, `wind_direction_deg`, `wind_gust_kph`
- `air_temp_c`

**Added at collection time:**
- `spot_id` (string), `lat`, `lon`
- `tide_height_m` — **always NaN** (see imputation below)
- `timestamp` — UTC datetime

**Computed during feature extraction (for model training):**
- `is_offshore`, `is_light_wind` (boolean flags)
- `wave_energy_proxy` = `swell_height_m² × swell_period_s`
- `wind_wave_interaction` = `wind_speed × cos(wind_dir − swell_dir)`
- `swell_wind_ratio`, `swell_dir_sin/cos`, `wind_dir_sin/cos`
- Cyclic time encodings: `hour_sin`, `hour_cos`, `month_sin`, `month_cos`, `day_of_week`
- `skill_level_encoded` (integer 0/1/2)

The model uses **26 features** in total (see `ml/models/model_metadata.json`).

### Synthetic labels

There are no real surf-quality labels. The dataset is a weather archive. Labels are **computed synthetically** by `ml/labels.py` using a physics-grounded formula from Scarfe et al. (2009).

The formula produces a score in [0, 100] from four additive components:

| Component | Max | Formula |
|-----------|-----|---------|
| Wave energy | 40 | `min(H² × T / 162, 1) × 40` — multiplicative, normalised against a 3 m / 18 s reference |
| Wind alignment | 25 | Cosine similarity of wind to the spot's offshore direction; penalty if speed > 25 kph |
| Tidal quality | 15 | Gaussian falloff centred on the spot's preferred tide height |
| Swell period | 20 | Linear ramp 0→10 for T < 10 s, then 10→20 for T in [10, 14 s], flat at 20 above 14 s |

This synthetic score is deliberately **structurally different** from the rule-based scorer in `app/agents/condition_agent.py` (which uses additive wave/wind/offshore components with weights 40/30/20/10). That difference is intentional: the ML model learns to approximate the synthetic target, not the heuristic, making the ML vs. rule comparison in the thesis meaningful rather than circular.

### Train / Validation / Test split

The split is **temporal (chronological), not random**, to prevent data leakage across time:

| Split | Fraction | Rows | Period |
|-------|----------|------|--------|
| Train | 70 % | 61,403 | oldest rows |
| Val | 15 % | 13,159 | middle period |
| Test | 15 % | 13,158 | most recent rows |

The boundary is computed globally across all spots sorted by timestamp. The 70/15/15 cut points are at row indices 61,403 and 74,562. A 5-fold `TimeSeriesSplit` is used during grid search to avoid leakage within the training set as well.

### Missing values and imputation

**Tide data is entirely missing.** `tide_height_m` is set to `NaN` for every row in `historical.parquet`. Open-Meteo does not provide tidal predictions, and integrating per-region tide APIs (WorldTides, local hydrographic offices) would have required separate API keys and non-uniform coverage across all five spots.

During training (`ml/train.py`), a `SimpleImputer` with `strategy="median"` is fit on the training set and applied to all splits. Because tide values are all NaN, the imputer substitutes the training-set median of the `tide_height` feature. In practice, the per-spot preferred tide midpoints (defined in `ml/labels.py` as `preferred_tide_m`: 0.20–0.70 m depending on spot) are used by the label formula when tide is NaN, so the model learns against a constant tide signal rather than real tidal variation.

This is documented as a known limitation in `ml/data/DATA_PROVENANCE.md` and the thesis.

**All other columns** are complete (no NaN in marine or weather variables from Open-Meteo, as confirmed at collection time). The imputer is saved to `ml/models/imputer.joblib` so it can be applied consistently at inference time.

### Outliers

No explicit outlier removal is performed. The `HistGradientBoostingRegressor` is inherently robust to outliers via its leaf-based tree structure. The `value_sanity` metric in the agent-level evaluation checks that all forecast values fall within physically plausible ranges (wave height 0–15 m, swell period 0–30 s, wind speed 0–200 kph), and all snapshot data passed these checks.

### Model performance

After grid search (`TimeSeriesSplit`, 5-fold, over max_iter × max_depth × learning_rate × min_samples_leaf):

- **Best hyperparameters:** `learning_rate=0.1`, `max_depth=7`, `max_iter=500`, `min_samples_leaf=10`
- **Cross-validation R²:** 0.923
- **Validation R²:** 0.970, **Validation MAE:** 1.50, **Validation RMSE:** 2.50 (score points out of 100)

### ML vs rule-based head-to-head on the held-out test set

Source: `evaluation/baseline_vs_ml.csv`. The rule-based scorer is `app/agents/condition_agent.py`; it applies a fixed additive formula (wave 40 / period 30 / wind-penalty 20 / offshore-bonus 10) without any spot-specific parameters or learned weights.

**Regression group** (how well the continuous score tracks the synthetic label):

| Metric | ML | Rule-based |
|--------|----|-----------|
| R² | **0.945** | 0.017 |
| MAE | **2.06** | 12.72 |
| RMSE | **3.59** | 15.16 |

**Classification group** (how well the four-category rating — ideal / suitable / challenging / unsafe — is assigned):

| Metric | ML | Rule-based |
|--------|----|-----------|
| Accuracy | **0.940** | 0.642 |
| F1 macro | **0.923** | 0.663 |

**Ranking group** (whether the ordering of hours by quality is preserved):

| Metric | ML | Rule-based |
|--------|----|-----------|
| Spearman ρ | **0.950** | 0.666 |

ML outperforms the rule-based baseline on all three metric groups. The rule-based R² near zero (0.017) indicates that the fixed-weight formula's continuous score has almost no linear relationship with the synthetic label on the test set — the formula uses different weights, no multiplicative energy term, and no spot-specific offshore direction, so the two scores are structurally misaligned even when both produce reasonable ratings.

Per-spot breakdown (ML only, from `baseline_vs_ml.csv`):

| Spot | R² | MAE | RMSE | Accuracy | F1 | Spearman |
|------|----|-----|------|----------|----|----------|
| Ericeira | 0.983 | 1.18 | 2.04 | 0.963 | 0.921 | 0.990 |
| Hossegor | 0.991 | 1.01 | 1.55 | 0.963 | 0.954 | 0.996 |
| Gold Coast | 0.837 | 2.28 | 3.62 | 0.952 | 0.669 | 0.843 |
| Pipeline | 0.825 | 3.17 | 4.94 | 0.857 | 0.902 | 0.886 |
| Jeffreys Bay | 0.704 | 2.65 | 4.51 | 0.964 | 0.676 | 0.775 |

Jeffreys Bay and Pipeline show lower R² than the Atlantic spots. Both have swell directions that are less well-represented in the ERA5-reanalysis-backed Open-Meteo hindcast and more variable bathymetric filtering effects, which the model cannot fully recover from with wind/swell features alone.

## Part 3 — How the System Was Tested

There are two separate evaluation harnesses. They answer different questions.

---

### 3.1 — Per-agent evaluation (`evaluation/agent_eval/`)

**Question answered:** Do the individual agents (research, forecast, condition, trip planning) produce internally correct, well-structured outputs?

**How it works:**

1. For each scenario, the snapshot is loaded and passed directly to each agent in isolation — no live API calls.
2. Each agent's output is passed to a deterministic scoring function in `evaluation/agent_eval/metrics.py`.
3. Results are written to `evaluation/agent_eval/results.csv`.

**Metrics per agent:**

| Agent | Metrics |
|-------|---------|
| Research agent | `field_completeness` (12 required fields present), `coordinate_validity`, `hazard_coverage`, `skill_coherence` |
| Forecast agent | `field_completeness` (8 sub-fields per record), `temporal_coverage` (records / expected hours), `value_sanity`, `wind_direction_presence` |
| Condition agent | `rating_validity` (fraction with valid rating word), `score_range_validity` (score in [0,100]), `reasoning_presence`, `safety_threshold_compliance` (unsafe hours flagged as unsafe), `rating_score_monotonicity` (ideal > suitable > challenging > unsafe by mean score) |
| Trip planning agent | `window_detection` (windows found when suitable hours exist), `window_score_ranking` (sorted descending), `suitable_hour_coverage` (fraction of suitable hours covered), `min_hours_respected` |

There is also an **orchestrator coherence evaluation** (`--orchestrator` flag) that checks: `tool_sequence_valid` (research called before forecast), `skill_level_passed_correctly` (correct skill propagated to assess_conditions), `unsafe_warning_present` (safety text in final response when unsafe hours exist), `top_window_mentioned` (top-ranked window's time appears in the response).

---

### 3.2 — LLM baseline evaluation (`evaluation/llm_baseline/`)

**Question answered:** How does SurfSense compare to GPT-4o-mini on the same surf forecast inputs, across five output-quality dimensions?

#### Input

Two inputs are used — one per system:

**SurfSense receives a natural-language user query.** The driver (`driver.py`) injects the snapshot into the orchestrator so that SurfSense calls its full agent pipeline (research → forecast → assess → plan) but uses the frozen snapshot data instead of hitting the live API. This ensures both systems are evaluated on identical forecast numbers. The user query is dynamically constructed, for example:

```
"I'm a beginner surfer. Please check the surf conditions at Praia do Guincho on 2026-05-08.
Rate each hour as ideal, suitable, challenging, or unsafe, flag any unsafe hours, and identify
the best surf windows. No need to ask for further details."
```

**GPT-4o-mini receives a structured prompt** built from `evaluation/llm_baseline/prompt_template.txt`. The prompt includes:
- The skill level and spot name
- A formatted table of the full forecast (one row per hour: timestamp | wave avg | swell height | swell period | wind speed | wind direction | is_offshore)
- Five explicit tasks: rate each hour, identify best windows with start/end times, cite numerical values, flag unsafe hours above the injected safety thresholds
- The unsafe thresholds explicitly in the prompt (e.g., wave > 2.25 m or wind > 22 kph for a beginner)

This is an **asymmetric evaluation by design**: SurfSense receives an open-ended natural-language request and must retrieve and structure the forecast data through its agents; GPT-4o-mini receives the data pre-formatted in a table with explicit safety thresholds in the prompt. GPT-4o-mini therefore has an informational advantage on the factual and safety dimensions.

#### Runs

Each system is queried **3 times per scenario** (3 independent LLM calls with temperature 0.7). This produces 6 output files per scenario: `runs/<scenario_id>/surfsense/run_{1,2,3}.txt` and `runs/<scenario_id>/gpt4o_mini/run_{1,2,3}.txt`.

Both systems use the same Azure OpenAI deployment (same endpoint, same model, same API version), configured via `AZURE_OPENAI_*` environment variables.

#### Output

Both systems produce free-text output. The outputs are saved as `.txt` files and scored automatically by `evaluation/llm_baseline/score.py`.

Sample SurfSense output for `guincho_24h` (beginner):
```
Here's the surf conditions report for Praia do Guincho on May 8, 2026 for a beginner surfer:

### Hourly Ratings
- 00:00 - Suitable
- ...
- 12:00 - Unsafe (Waves too high and wind too strong)
...

### Best Surf Window
- 00:00 - 10:00: 11 hours of suitable conditions, with the peak score at 03:00 (60.8).
```

Sample GPT-4o-mini output for the same scenario:
```
### Surf Session Ratings at Praia do Guincho

| Hour (UTC) | Wave Height (m) | Wind Speed (kph) | Rating |
|...         | 0.8             | 3.8              | Ideal  |
...
### Best Continuous Surfing Window
- Start: 2026-05-08T00:00
- End:   2026-05-08T09:00
```

#### Evaluation dimensions (five-dimension rubric in `score.py`)

Each run is scored on five dimensions. The mean across 3 runs gives the scenario-system score. Some dimensions return `N/A` when they are not applicable (e.g., no unsafe hours exist in the snapshot).

| Dimension | What it measures | How it is computed |
|-----------|------------------|--------------------|
| **factual_consistency** | Numerical claims in the output match the forecast within 10 % | Regex extracts unit-suffixed numbers from prose (e.g. "1.5 m", "12 kph"). Each extracted value is compared against the nearest forecast value of the same unit. Values that match the prompt's injected safety thresholds are excluded (echoing the prompt is not a factual claim). Score = correct / total. |
| **safety_enforcement** | Unsafe hours are explicitly flagged in the output | Counts the number of hours in the snapshot that exceed 1.5× the skill-level thresholds (wave height or wind speed). Counts occurrences of the word "unsafe" in the output. Score = flagged / unsafe_hours. Returns N/A if no hours are unsafe. |
| **temporal_optimisation** | An explicit surf window with both start and end times is identified | Looks for an inline time range ("12:00 to 16:00", "6am - 10am") or a labelled pair ("Start: X ... End: Y" within 5 lines). A bare list of timestamps does not count. Returns N/A if the snapshot has no surfable hours. |
| **consistency** | The three runs agree with each other | Mean pairwise normalised Levenshtein similarity: `1 − edit_distance(A, B) / max(len(A), len(B))` across all 3 pairs. Only valid (non-error) runs are included. |
| **explainability** | Ratings are accompanied by specific numbers | For each line containing a rating word (ideal/suitable/challenging/unsafe), checks whether that line or the two following lines contain a unit-suffixed number. Score = fraction of ratings with a number nearby. |

#### Results summary

Results are stored in `evaluation/llm_baseline/results.csv`. Each row is one (scenario, system, dimension) triple, with per-run scores in `run_1/run_2/run_3` columns and the mean in `score`. A value of "N/A" means the metric was not applicable for that snapshot.

Key patterns visible in the results:
- Both systems achieve **factual_consistency ≈ 1.0** on almost all scenarios.
- GPT-4o-mini achieves **temporal_optimisation = 1.0** consistently; SurfSense is more variable (often 0.33 mean across 3 runs, meaning only one of the three runs identified an explicit window).
- SurfSense tends to score higher on **explainability** than GPT-4o-mini in most scenarios.
- **Safety_enforcement** returns N/A on many scenarios because most snapshots do not contain genuinely unsafe hours (the unsafe threshold is 1.5× the base skill ceiling, which is a high bar).
- **Consistency** is higher for GPT-4o-mini on average; SurfSense outputs show more variation across runs because the full agent pipeline introduces more stochastic intermediate steps.
